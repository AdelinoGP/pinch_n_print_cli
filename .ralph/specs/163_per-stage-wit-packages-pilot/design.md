# Design: 163_per-stage-wit-packages-pilot

## Controlling Code Paths

- Primary code path: the stage→package→typed-instantiate chain. `crates/slicer-schema/wit/deps/<pkg>/<pkg>.wit` (contract) → `slicer_schema::STAGES` (the sole lookup table, ADR-0006) → `crates/slicer-wasm-host/src/host.rs`'s `bindgen!` mods (host bindings) → `crates/slicer-wasm-host/src/dispatch.rs`'s three pilot runner fns (instantiate + call) → `crates/slicer-macros/src/lib.rs`'s glue builders (guest side) → `xtask/src/build_guests.rs` (which guests must rebuild).
- Neighboring tests/fixtures: `crates/slicer-runtime/tests/contract/{wit_single_source_tdd.rs, wit_drift_detection_tdd.rs, dispatch_protocol_tdd.rs, postpass_gcode_*_tdd.rs, macro_postpass_text_roundtrip_tdd.rs}`; `crates/slicer-runtime/tests/executor/{postpass_executor_tdd.rs, finalization_*_tdd.rs}`; `crates/slicer-macros/tests/{binding_surface_tdd.rs, all_worlds_glue_tdd.rs, slicer_module_tdd.rs}`; `crates/slicer-schema/tests/export_for_stage_id_tdd.rs`; guests `crates/slicer-wasm-host/test-guests/{postpass-guest, sdk-postpass-text-guest, finalization-guest, finalization-mutation-roundtrip-guest, sdk-finalization-guest}`.
- OrcaSlicer comparison: **none, and deliberately so.** This is a host/contract refactor of a WASM component module system. OrcaSlicer is a monolithic C++ binary with no module system, no WIT, and no plugin ABI — there is no canonical behavior to port or match, and ADR-0045 itself cites OrcaSlicer only to argue a *negative* (it has no lifecycle hook), which packet 162 already consumed. The `orca-delegation` snippet is therefore omitted from `packet.spec.md` and `requirements.md` rather than filled with a fictional path.

## Architecture Constraints

- **The unit of versioning is the package, not the interface.** WIT attaches `<semversuffix>` to `package-name`; there is no interface-version production. Proof in-tree: `crates/slicer-schema/wit/deps/common.wit` declares `package slicer:common;` (unversioned) holding `module-errors` and `host-services`, neither of which can carry a version. Grouping stages as sibling interfaces inside one tier package would make a breaking change to one stage major the package and move every sibling's wasmtime alt-key — ADR-0045 §"The unit is the package" replays packet 130 to show it fails on the ADR's own motivating case. One package per stage is the decision; do not re-litigate it.
- **Every package starts at `@1.0.0`, never `0.x` — mechanically load-bearing, not housekeeping.** `wasmtime-environ-43.0.1/src/component/names.rs::alternate_lookup_key` yields a major-track key only for major ≥ 1 (`x:y/z@1.1.2 → x:y/z@1`); at `@0.1.0` you get the minor-track `@0.1` so every minor bump breaks, and at `@0.0.1` you get `None`. ADR-0045 §"Verified empirically, not just read" **measured** all five rows on the pinned toolchain: `guest@1.0.0/host@1.0.0` OK; `guest@1.0.0/host@1.5.0` OK (the load-bearing row — the alt-key `@1` resolves); `guest@1.5.0/host@1.0.0` OK (both directions); `guest@1.0.0/host@2.0.0` FAIL; `guest@0.1.0/host@0.2.0` FAIL. Failures are clean, at `instantiate`, before any call. AC-1b pins `major >= 1` mechanically so a later "tidy-up" to `0.x` cannot silently disable every compatibility claim.
- **The engine's miss diagnostic is expected-only.** wasmtime emits `` no exported instance named `slicer:x/y@2.0.0` `` — it names what the **host wanted** and never what the **guest shipped**. ADR-0045's prose ("a diagnostic naming the expected `package/iface@version`") is satisfied by the engine verbatim and no more. This packet therefore **does not** budget host-side decoding of the component's actual exports to produce "expected @2.0.0, found @1.0.0"; AC-N1 asserts the engine's own wording and explicitly forbids asserting a "found" fragment the engine does not produce. If a future packet wants the richer message it is real host-side work against `wasmtime::component::Component`'s export list, and it needs its own step.
- **No probing, fatal on miss.** `[stage] id` is singular in all 20 core manifests (verified: each `modules/core-modules/*/*.toml` declares exactly one `[stage] id`), so dispatch always knows which package to instantiate. ADR-0045 §"Alternatives rejected" drops the original "probe each and tolerate the miss" rule precisely because tolerating a miss recreates the silent-success failure this work exists to delete.
- **`[stage] id` stays in the manifest.** The DAG validator and `dag_cli` plan without instantiating any WASM (ADR-0006's rejected alternative), so the stage must be declarable ahead of load.
- **One Rust type set across worlds (ADR-0002 / packet 75).** Each non-`layer` `bindgen!` mod aliases the layer world's generated dep types via `with:`. Every new per-stage mod must repeat that block verbatim — see §"Data and Contract Notes".
<!-- snippet: wasm-staleness -->
- Guest WASM is **not** rebuilt by `cargo build` or `cargo test`. After editing any path in this packet's change surface that feeds the guest build (see `CLAUDE.md` §"Guest WASM Staleness"), the implementer MUST run `cargo xtask build-guests --check` and, if `STALE:` is reported, rebuild without `--check` before re-running the failing test. Stale-guest failures look unrelated to the change but are caused by it.

## Code Change Surface

### Selected approach

The package/interface/world skeleton is copied from the one ADR-0045 §"Verified empirically, not just read" **executed** against the pinned wasmtime 43.0.1 / wit-bindgen 0.57.1 — `package spike:alpha@1.0.0; interface foo { … } world alpha-module { export foo; }`, host `bindgen!({ path: …, world: "spike:alpha/alpha-module" })` + `inst.spike_alpha_foo().call_hello(…)`, guest `impl exports::spike::alpha::foo::Guest`. Do not invent a different one. The build path is unchanged: `cargo build --target wasm32-unknown-unknown --release` then `wasm-tools component new`, exactly the two steps `xtask/src/build_guests.rs::build_one` already runs.

**One deviation from the spike's shape is mandatory, and it is the packet's central design decision.** The spike's `foo` had no parameters. Our stages take **host-owned resources** as parameters (`gcode-output-builder`, `layer-collection-view`, `finalization-output-builder`), and today those are declared in the **world body** of `world-postpass.wit` / `world-finalization.wit`, which makes them host-implemented — `crates/slicer-wasm-host/src/host.rs::postpass_impls` really does `impl ppm::HostGcodeOutputBuilder for HostExecutionContext`, and `finalization_impls` does the same for `LayerCollectionViewData` / `FinalizationOutputBuilderData`. A resource declared inside an **exported** interface is owned by the **guest** (wit-bindgen generates `exports::…::GuestGcodeOutputBuilder` for the guest to implement). Naively folding the world body into the exported interface would therefore invert ownership and break every host builder impl. The resources must live in an **imported** interface — which is precisely what `world-layer.wit` already does, taking its `perimeter-output-builder` / `infill-output-builder` from the imported `slicer:ir-handles/ir-handles`.

So each resource-bearing stage package holds **two** interfaces — one imported, one exported — in the same package, hence at one version, hence still exactly one version per stage:

```wit
// crates/slicer-schema/wit/deps/postpass-gcode-postprocess/postpass-gcode-postprocess.wit
package slicer:postpass-gcode-postprocess@1.0.0;

// IMPORTED. Holds the host-owned resource and the plain data types.
// These must not live in the exported interface: a resource in an exported
// interface is implemented by the GUEST, which would invert today's
// `impl ppm::HostGcodeOutputBuilder for HostExecutionContext`.
interface gcode-postprocess-types {
    use slicer:types/geometry.{extrusion-role};

    record gcode-move-cmd { ... }          // moved verbatim from world-postpass.wit
    variant retract-mode { gcode, firmware }
    record gcode-retract-cmd { ... }
    record gcode-fan-speed-cmd { ... }
    record gcode-temperature-cmd { ... }
    record gcode-tool-change-cmd { ... }
    resource gcode-output-builder { ... }
    variant gcode-command { ... }
}

// EXPORTED. Holds the stage entry point and nothing else.
interface gcode-postprocess {
    use gcode-postprocess-types.{gcode-command, gcode-output-builder};
    use slicer:config/config-types.{config-view};
    use slicer:common/module-errors.{module-error};

    run: func(
        commands: list<gcode-command>,
        output: gcode-output-builder,
        config: config-view,
    ) -> result<_, module-error>;
}

world gcode-postprocess-module {
    import slicer:common/host-services;
    import slicer:config/config-types;
    import gcode-postprocess-types;
    export gcode-postprocess;
}
```

`postpass-text-postprocess` needs **no** `-types` interface — `run: func(gcode-text: string, config: config-view) -> result<string, module-error>` takes no resource beyond the already-imported `config-view`, so it is a single exported interface. The asymmetry is real and correct; do not add an empty `text-postprocess-types` for symmetry. `finalization-layer-finalization` follows the two-interface shape: imported `layer-finalization-types` holds `layer-idx`, `object-id`, `region-id`, `region-key`, `tool-change-view`, `print-entity-view`, `z-hop-view`, `entity-mutation`, `sort-key`, `synthetic-layer-data`, `resource layer-collection-view` and `resource finalization-output-builder` — all moved verbatim from `world-finalization.wit`; exported `layer-finalization` holds only `run: func(layers: list<layer-collection-view>, output: finalization-output-builder, config: config-view) -> result<_, module-error>`. Because the type bodies move verbatim, `dispatch.rs::apply_finalization_pushes` and `host.rs::finalization_impls` change only in module path.

### Naming — the `[DECIDE + FLAG]` resolved

**The rule is `slicer:<tier>-<stage-name-in-kebab-case>@1.0.0`**, where `<tier>` is the **WIT tier** (`layer`, `prepass`, `postpass`, `finalization`) and `<stage-name>` is the `StageSpec.stage_id` local part — the segment after `::` — kebab-cased. It reproduces all **16** names with **no exception**, and it derives each package from the stage's identity rather than from its export spelling.

| stage_id | tier | package |
|---|---|---|
| `PostPass::GCodePostProcess` | postpass | `slicer:postpass-gcode-postprocess@1.0.0` |
| `PostPass::TextPostProcess` | postpass | `slicer:postpass-text-postprocess@1.0.0` |
| `PostPass::LayerFinalization` | **finalization** | `slicer:finalization-layer-finalization@1.0.0` |
| `Layer::Perimeters` | layer | `slicer:layer-perimeters@1.0.0` (matches ADR-0045) |
| `Layer::InfillPostProcess` | layer | `slicer:layer-infill-postprocess@1.0.0` (matches ADR-0045) |
| `PrePass::SeamPlanning` | prepass | `slicer:prepass-seam-planning@1.0.0` (matches ADR-0045) |

Two things the rule must not be confused with:

- **The tier is the WIT tier, not the `stage_id` prefix.** `PostPass::LayerFinalization` lives in the *finalization* tier despite its `PostPass::` prefix (`StageSpec.world_id == WORLD_FINALIZATION`). Read the tier from `world_id`, never by splitting `stage_id`. That quirk is in the tree and is not "fixed" here.
- **Not `<tier>-<wit_export minus "run-">`.** That variant reproduces 15 of 16 and degenerates on the finalization tier, whose live export is `run-finalization` → `slicer:finalization-finalization`. Deriving from the stage name instead removes the exception rather than managing it. The export spelling is the wrong source anyway: this packet repurposes `wit_export` to `"run"`, so a rule keyed on it would stop being computable the moment it was applied.

`slicer:finalization@1.0.0` was considered and rejected. It re-encodes "this tier has exactly one stage" into package identity — the precise tier-as-contract conflation ADR-0045 exists to end ("The tier survives as **vocabulary**; it dies as a **contract**"). The moment a second finalization stage lands, `slicer:finalization` must either be renamed — breaking all four finalization core modules and any out-of-tree module — or become a mis-named tier package holding two stages, which is the rejected alternative "stages as interfaces inside one tier package". Under the rule, adding `slicer:finalization-<new>@1.0.0` is purely additive. `finalization-layer-finalization` reads as a stutter only because the tier and the stage genuinely share a word; it is the rule applied without special-casing, not an exception to it. Flagged `[FWD]` because packet #3 must apply the same rule to the 13 remaining stages.

### The func is `run`, not `run-gcode-postprocess`

ADR-0045's decision block is explicit: `interface perimeters { run: func(...) -> result<_, module-error>; }`. The stage name already lives in the package and the interface, so `slicer:postpass-gcode-postprocess/gcode-postprocess@1.0.0#run` is the idiomatic component-model spelling (cf. `wasi:cli/run@0.2.0#run`); repeating it a third time in the func stutters across all seventeen packages. Consequence: `StageSpec.wit_export` becomes `"run"` for the three pilot rows, so `export_for_stage_id` returns `"run"` for them. Its own guard (`crates/slicer-schema/tests/export_for_stage_id_tdd.rs::export_for_stage_id_is_total_over_stages_and_rejects_unknown`) and `crates/slicer-runtime/tests/contract/dispatch_protocol_tdd.rs:138` both compare against the table, so they stay green by construction — the churn is the literal `"run-gcode-postprocess"` / `"run-finalization"` in the five pilot modules' `slicer_module_binding_tdd.rs` and in `crates/slicer-macros/tests/`. The descriptive identity is not lost: `qualified_export_for_stage_id` reconstructs `slicer:postpass-gcode-postprocess/gcode-postprocess@1.0.0#run` from the same row, and that is what the fatal-on-miss diagnostic prints.

### Rejected alternatives and reasons

- **Keep the func named `run-gcode-postprocess` inside `interface gcode-postprocess`.** Minimises churn (`wit_export` untouched, 5 binding tests unchanged), but contradicts ADR-0045's decision block, and stutters the stage name three times in one export identity across all 17 packages. The churn it avoids is 5 one-line edits.
- **Leave `xtask/src/build_guests.rs` alone and prove isolation only at the binary level** (rebuild everything, then assert `arachne-perimeters.wasm` decodes no pilot package). Rejected: ADR-0045's headline row is literally "untouched, **doesn't even rebuild**", and `compute_shared_mtime`'s `wit/**/*.wit` max means that claim is false today no matter how the packages are cut. Deferring the fix would ship a pilot that cannot demonstrate the thing it is piloting, and would hand packet #3 the same hole.
- **Give test guests a stage via `[package.metadata.slicer] stage_id` in their `Cargo.toml`.** Would give full per-stage granularity, but costs 12 file edits for no correctness gain — conservative test guests over-rebuild, which is safe and never wrong. `[FWD]` to #3.
- **Add a composite `world both { export gcode-postprocess; export text-postprocess; }` in `postpass-guest/wit/` so one guest keeps covering both stages.** It would work (a component may export more interfaces than the world requires, so the host could still instantiate either single-stage world against it), and it would demonstrate multi-package export — but it depends on `wit_bindgen::generate!` resolving a `path:` list, which is unverified here, and it adds a second WIT authoring location for zero test coverage gained. Rejected in favour of gcode-only.
- **Fold the world body straight into the exported interface** (the naive reading of ADR-0045's `interface foo { hello: func() -> string; }` skeleton). Rejected because the spike's `foo` takes no parameters while our stages take host-owned resources: a `resource` in an exported interface is guest-implemented, which inverts `impl ppm::HostGcodeOutputBuilder for HostExecutionContext`. Hence the imported `<iface>-types` companion. AC-1 asserts zero `resource` declarations in any exported interface so this cannot be reintroduced.
- **Enrich the miss diagnostic to "expected @1.0.0, found @2.0.0".** ADR-0045's spike measured that wasmtime supplies expected-only (`` no exported instance named `slicer:x/y@2.0.0` ``). Producing "found" means decoding the component's export list host-side — real work, not free. Rejected here: the engine's wording already satisfies the ADR's promise, and the pilot should move one variable. AC-N1 forbids asserting a message the engine does not produce.
- **`with:`-map the moved resources onto `PostpassGcodeOutputBuilderData` / `LayerCollectionViewData` / `FinalizationOutputBuilderData`.** Tempting while the file is open, but today they are unmapped bindgen markers and the packet's risk is already concentrated in resource identity across 5 `bindgen!` calls (AC-12). Changing the mapping at the same time would make an AC-12 failure ambiguous between the split and the remap.
- **Migrate all 17 stages in one packet.** The plan's queue splits it deliberately: this packet is the machinery plus 3 exports; #3 is 13 exports with no new machinery. A single packet would be L.

## Exports handed to packet #3

`164_per-stage-wit-packages-bulk` (TASK-146c) migrates the remaining 13 stages using **only** the machinery below. Consume these; do not re-derive them, and do not add a parallel table.

Net-new in `slicer_schema` (`crates/slicer-schema/src/lib.rs`):

- `StageSpec.wit_dir: &'static str` — the directory under `crates/slicer-schema/wit/deps/` holding this stage's package. **Total over all 16 rows**: the package dir for a migrated stage, `"world-layer"` / `"world-prepass"` for an unmigrated one. #3 flips the remaining 13 to their package dirs and deletes the two tier values.
- `StageSpec.wit_package: &'static str` — e.g. `"slicer:postpass-gcode-postprocess@1.0.0"`. `""` while unmigrated.
- `StageSpec.wit_interface: &'static str` — e.g. `"gcode-postprocess"`. `""` while unmigrated.
- `StageSpec.wit_world: &'static str` — e.g. `"gcode-postprocess-module"`. `""` while unmigrated.
- `StageSpec.wit_export` — **repurposed**: `"run"` for a migrated stage; still the old `"run-<stage>"` spelling for the 13 unmigrated ones. #3 sets the rest to `"run"`.
- `package_for_stage_id(&str) -> Option<&'static str>`, `interface_for_stage_id(&str) -> Option<&'static str>`, `wit_world_for_stage_id(&str) -> Option<&'static str>`, `wit_dir_for_stage_id(&str) -> Option<&'static str>` — `None` for an unknown stage; the first three also `None` for an unmigrated one.
- `qualified_export_for_stage_id(&str) -> Option<String>` — the full component-model export identity, e.g. `"slicer:postpass-gcode-postprocess/gcode-postprocess@1.0.0#run"`. `None` for unknown **and** unmigrated stages. This is the only lookup that fully identifies a migrated stage's contract; `export_for_stage_id` no longer does.

## Why `postpass_gcode_boundary` is not in AC-12

AC-12 lists four round-trip suites. A fifth, `crates/slicer-runtime/tests/contract/postpass_gcode_boundary_tdd.rs::*`, looks like it belongs and **must not be added**. It cannot detect anything AC-12 exists to detect:

- Its helper takes `_component: Arc<slicer_runtime::WasmComponent>` (`:63`) and **discards it** — the leading underscore is a silenced unused-variable warning.
- It calls the dispatcher with `WasmInstancePool::placeholder()` and `None` (`:190-191`).
- In `dispatch.rs`, that `None` returns `Err(DispatchError { phase: DispatchPhase::MissingComponent, .. })` (`:1014` for the gcode runner) **before** any linker, store, resource push, or `instantiate`.
- That error is then converted to success by `Err(e) if e.phase == DispatchPhase::MissingComponent => Ok(PostpassOutput::GCodeSuccess)` (`dispatch.rs:1721`, `:1925`, `:1970`, `:2017` — four sites, none behind `#[cfg(test)]`).
- The test asserts `matches!(result, Ok(PostpassOutput::GCodeSuccess))` and `gcode_ir.commands == expected` (`:199-200`), where `expected` is the exact vec it fed in.

No WASM executes. It passes with the guest deleted, and it would pass with `with:`-mapped resource identity completely broken — the precise failure AC-12 is the falsifier for. Including it would have made the packet's headline AC partly vacuous.

**Do not overstate this, and do not "repair" it.** The skip is **deliberate, not a bug**: `crates/slicer-wasm-host/src/binding.rs:29` documents `CompiledModuleLive::wasm_component: Option<Arc<WasmComponent>>` as "`None` for placeholder or test modules", and the four conversion sites are load-bearing for host-builtin stages that legitimately have no component. The **proven** defect is only the vacuous test. Whether a *real* module whose component failed to load can also reach `None` and be silently skipped — which would be an ADR-0015 "do not swallow" violation — is **untraced**, and this packet asserts nothing about it either way.

It is already filed and committed as **`DEV-087`** (`57ceae39`), which carries this same chain and marks the production question as untraced rather than asserting it. **Reference `DEV-087`; do not file a second row.** This packet only *stops relying on* the test. The four suites AC-12 does list were each verified to assert on a value only a live guest can produce, and none is `#[ignore]`d.

## Data and Contract Notes

- **IR/manifest contracts:** unchanged. `[stage] id` stays singular; `wit-world` stays in all 20 manifests (AC-10). No IR schema, config key, or claim moves. Config key strings stay snake_case per `CLAUDE.md`.
- **WIT boundary — what per-stage bindgen means for ADR-0002's type sharing. This is the packet's central risk and its reason to exist.** ADR-0045's spike proved the packaging mechanism with **no host imports at all**. The real stage worlds import `slicer:common/host-services`, `slicer:config/config-types` and (transitively) `slicer:ir-handles/ir-handles`, and rely on `with:`-mapped resource **identity** holding across separate `bindgen!` calls: ADR-0002 remaps every non-layer world onto the layer world's single Rust type set, and ADR-0003 explains why guests cannot share the same way. Nothing measured so far tests that. Splitting 4 `bindgen!` calls into 5 (and, at packet #3, 17), each `with:`-mapping the same dep interfaces, is the untested part; if identity does not hold you get exactly the linking failure `CLAUDE.md` §"WIT/Type Changes Checklist" warns about (`list<object-id>` in one file vs `list<MeshObjectView>` in another). **163 exists to prove the mechanism survives contact with real imports and `with:`-mapped resources on the cheapest tier, before #3 commits the other 13 stages.** AC-12 is the falsifier and must stay a *real typed instantiation plus a real host↔guest resource round-trip*, never a decode-only check — a decode check passes even when identity is broken.
  Grounded specifics: the two tier worlds being split define their types **inline in the world body**, not in a shared interface, so those types are already **not** shared with the layer world — the `with:` block of `pub mod postpass` (`host.rs:611-621`) aliases only the five dep packages `slicer:types/geometry`, `slicer:config/config-types`, `slicer:common/host-services`, `slicer:common/module-errors`, `slicer:ir-handles/ir-handles`. Consequences: **(a)** yes, each of the three new stage worlds needs the full `with:` block, and it is the *same* five keys, copied verbatim — the churn is 3 × 5 lines, not a redesign; **(b)** the alias for `slicer:ir-handles/ir-handles` is required even though neither postpass world names it, because `common.wit`'s `host-services#generate-arachne-walls` does `use slicer:ir-handles/ir-handles.{extrusion-line}` (packet 112 Step 9A) and every world importing `host-services` pulls it transitively — the same reason `emit_world_preamble` nests `IR_TYPES_WIT` unconditionally; **(c)** the moved resources stay **unmapped** by `with:`, exactly as today: `postpass_impls` uses the bindgen-generated marker (`Resource<ppm::GcodeOutputBuilder>`) and `HostExecutionContext::push_postpass_gcode_output_builder` pushes `PostpassGcodeOutputBuilderData` into the `ResourceTable`. Do not add `with:` entries for them — that is a second change riding on the first, and the point of the pilot is to move one variable; **(d)** the generated Rust type paths shift from `host::postpass::GcodeCommand` to `host::postpass_gcode::slicer::postpass_gcode_postprocess::gcode_postprocess_types::GcodeCommand` (an *imported* interface, hence no `exports::` segment) — mechanical, and confined to `dispatch.rs`'s two converter fns plus `postpass_impls` / `finalization_impls`; **(e)** a guest whose `wit/` tree resolves all 17 packages while exporting one costs nothing at runtime — ADR-0045's spike measured a guest whose tree contained a deliberately-broken `beta@2.0.0` exporting only alpha and instantiating fine — so keeping `root.wit` as the umbrella anchor and adding `deps/<stage>/<stage>.wit` alongside the existing unversioned dep packages is safe at #3's scale.
- **Nested vs. per-package `deps/` — grounded, both answers differ by consumer, and both syntaxes are measured to work.** ADR-0045 §"Verified empirically" confirms both layouts resolve on host and guest: `deps/<pkg>/<pkg>.wit` directories, and nested `package foo:a@1.0.0 { }` blocks in one file (guest, via `inline:`). We use each where it already fits. *Host:* `bindgen!({ path: "../slicer-schema/wit", … })` hands wit-parser a **directory**, which resolves `root.wit` as the top package and every entry under `deps/` as a dependency package. Each new stage package therefore needs its own `deps/<name>/<name>.wit` subdirectory — the same layout the four tier worlds already use — and `path:` stays `"../slicer-schema/wit"` for all five mods (`wit_single_source_tdd.rs::host_bindgen_paths_target_shared_root` enforces exactly that: no `path:` may contain a `/world-` segment). *Guest:* `crates/slicer-macros/src/lib.rs::emit_world_preamble` does **not** use `path:`; it `include_str!`s the world file as the top-level statement-form `package …;` header and brace-wraps each dep as a nested `package slicer:X { … }` block, feeding the concatenation to `wit_bindgen::generate!{ inline: … }`. Multiple packages in one blob therefore already work here — nested brace form, exactly one statement-form header. A per-stage package file is a drop-in replacement for the world file in that assembly; no guest-side `deps/` dir is needed and none is added.
- **Interface-grouped exports change the macro's emission shape.** A world exporting a bare func generates `trait Guest { fn run_gcode_postprocess(...) }` and `export!(C)`. A world exporting an *interface* generates `exports::slicer::postpass_gcode_postprocess::gcode_postprocess::Guest` with `fn run(...)`, and `export!(C)` still applies. So `build_postpass_gcode_glue` emits `impl exports::slicer::postpass_gcode_postprocess::gcode_postprocess::Guest for __SlicerPostpassGcodeComponent { fn run(...) { … } }`. This is the shape AC-6 pins.
- **Determinism/scheduler constraints:** none introduced. Dispatch order, DAG planning and `STAGE_ORDER` are untouched; `[stage] id` remains the planner's input and no WASM is instantiated during planning.
- **The fatal-on-miss diagnostic is not a new mechanism.** `DispatchError` already models `DispatchPhase::TypedInstantiation`, and wasmtime's `Indices::new` already fails there when an export is absent. Today the failure never happens for postpass/finalization because the macro's padding guarantees the exports exist. Deleting the padding is what makes the existing error path reachable; the packet only enriches its `reason`.

## Locked Assumptions and Invariants

- **Package versions are locked at `@1.0.0` and are load-bearing.** They are not cosmetic: they are what makes wasmtime's `alternate_lookup_key` produce a `@1` major-track key. A future contributor "tidying" them to `@0.1.0` silently breaks every minor-version compatibility claim. `wit_drift_detection_tdd.rs` must pin the exact `@1.0.0` string per package.
- **Package/interface/world names are a public contract** the moment a third party compiles against them. ADR-0045 §Consequences is explicit that breaking changes are free now and will not be later. `slicer:finalization-layer-finalization` is chosen (over `slicer:finalization`) precisely so that adding a second finalization stage stays additive.
- **`wit_export == "run"` for migrated stages** — `export_for_stage_id` alone no longer identifies a stage; `qualified_export_for_stage_id` does. Any new consumer must use the qualified form.
- **Two contract mechanisms are live in-tree between this packet and #3.** Accepted, recorded as `DEV-086` in `docs/DEVIATION_LOG.md` with #3 named as owner. `docs/specs/adr-0045-per-stage-wit-packages-plan.md` §"Dependency note" states why #3 is not optional: leaving it undone reproduces the exact failure this ADR exists to end (cf. `run.rs`'s 2026-05 "pragmatic fix", still load-bearing 14 months later).
- **`xtask` staleness is conservative by construction.** An unknown stage (every test guest) depends on all package dirs. Over-rebuilding is safe; under-rebuilding is the bug class `CLAUDE.md` §"Guest WASM Staleness" exists to prevent. Never invert this default to make an AC pass.
- Reversibility: the WIT split is a breaking contract change with no config flag. Reverting means reverting the packet.

## Risks and Tradeoffs

- **Resource identity across 5 `bindgen!` calls is unproven and is the packet's headline risk** (ADR-0045's spike used no imports at all). If `with:`-mapped identity does not survive the split, the failure is a compile/link error at Step 6 or an AC-12 failure, not a subtle one — and it lands on the cheapest tier, with 13 stages still unmigrated, which is exactly why the pilot is scoped this way. If AC-12 cannot be made green, **stop and return the finding rather than widening scope**: it invalidates packet #3's premise, not just this packet's code.
- **The imported `<iface>-types` companion interface is a shape neither ADR-0045's prose nor its spike covers.** It is derived here from the in-tree fact that `postpass_impls`/`finalization_impls` host-implement those resources today, and from `world-layer.wit`'s precedent of taking its builders from the imported `slicer:ir-handles/ir-handles`. Packet #3 inherits the shape for the 10 layer stages — where the resources already live in an imported dep interface, so most layer packages will need **no** `-types` companion at all. Confirm before generalizing.
- **The tree does not compile between Step 1 and Step 6.** Six layers move together. Mitigation: `implementation-plan.md` fixes the order, and no step before 6 lists a `cargo check` exit. Risk: an implementer reads a red `cargo check` at Step 3 as a defect and starts "fixing" out-of-scope files.
- **`compute_shared_mtime` currently defeats the ADR's headline claim** (grounded, see `requirements.md` §Problem Statement). Absorbing the `xtask` fix grows the packet by one step. Not absorbing it makes AC-N2 unprovable.
- **AC-N2 is mtime-sensitive.** It `touch`es a `.wit` after a full rebuild and depends on filesystem timestamp granularity; the command inserts `sleep 2` before the `touch` for that reason. It also leaves guests stale — re-run `cargo xtask build-guests` afterwards.
- **`postpass-guest` loses `run_text_postprocess`.** Verified safe against every call site today (all five use it for gcode only; all three text round-trips use `sdk-postpass-text-guest`). If a future text test wants a hand-written oracle it needs a new guest, not this one.
- **`resolve_stage_glue`'s deleted trait fallback changes behavior for a stageless `impl PostpassModule`/`impl FinalizationModule`.** Today it silently emits a fully-padded component; after this packet it emits no component glue. That is the intended direction (a stageless impl has no stage and therefore no package), but if any test or module relies on the old fallback it will surface at Step 6. Verified: all 20 core manifests declare exactly one `[stage] id`, so no shipped module depends on it.
- **`slicer-schema` becomes an `xtask` build dependency.** Trivial (leaf lib, no deps), but it means a `slicer-schema` compile error now also breaks `cargo xtask build-guests` — the very command used to diagnose such failures. Acceptable: the alternative is duplicating the stage→dir table in `xtask`, which ADR-0006 forbids.
