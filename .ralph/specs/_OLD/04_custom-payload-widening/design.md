# Design: 04_custom-payload-widening

## Controlling Code Paths

- **WIT type definitions (canonical source)**: `wit/deps/types.wit` (for `extrusion-role`) and `wit/deps/ir-types.wit` (for `paint-semantic` and `wall-feature-flag`). These are the canonical disk files established by Packet A. All converters must match these types.
- **Macro converters**: `crates/slicer-macros/src/lib.rs` — converter functions:
  - `__slicer_ir_role_to_wit` (~line 1774): converts IR `ExtrusionRole` → WIT `extrusion-role`
  - `__slicer_wit_semantic_to_ir` (~line 1619): converts WIT `paint-semantic` → IR `PaintSemantic`
  - `__slicer_wit_feature_to_ir` (~line 1589): converts WIT `wall-feature-flag` → IR `WallFeatureFlags`
  - `__slicer_ir_feature_to_wit` (~line 1822): converts IR `WallFeatureFlags` → WIT `wall-feature-flag`
  - Also `ir_to_wit_paint_semantic` (~line 1406) and `__slicer_ir_semantic_to_wit` (around line 1628): paint semantic conversion functions used in the layer-world glue
- **Host converters**: `crates/slicer-host/src/wit_host.rs` — search for `String::new()` synthesis near the converter functions. There are 5 known sites:
  1. `finalization_world` glue (~line 788): `ExtrusionRole::Custom` → `Custom(String::new())`
  2. `layer_world` `__slicer_wit_role_to_ir` (~line 1566): `WitExtrusionRole::Custom` → `Custom(String::new())`
  3. `__slicer_wit_semantic_to_ir` (~line 1625): `WitPaintSemantic::Custom` → `Custom(String::new())`
  4. `finalization_role_wit_to_ir` (~line 2285): `fgeo::ExtrusionRole::Custom` → `Custom(String::new())`
  5. `convert_extrusion_role` (~line 2563): `ExtrusionRole::Custom` → `Custom(String::new())`
  - Additionally, `convert_wall_feature_flag` (~line 2817) has no `custom` field yet because the WIT field does not exist. After adding the WIT field, this converter must also be updated.
- **WIT encode/decode trait implementations**: `wit-bindgen` generates trait implementations for the WIT types. The macro provides hand-written trait impls for the converter functions that bridge between IR types and WIT types. These impls must be updated to handle the new variant shapes.

## Architecture Constraints

- **WIT representation for `WallFeatureFlags.custom`**: `list<tuple<string, paint-value>>`. wasmtime's component model bindgen supports this directly. Encode `HashMap<String, PaintValue>` as this list and decode it back. Sort by key for determinism.
- **WIT variant vs enum**: Changing `enum` to `variant` in WIT is a breaking change for `wit-bindgen`-generated types. The macro's hand-written converter trait impls must be updated to match the new variant structure. Host bindings (generated by `bindgen!`) will update automatically when recompiled.
- **WIT `paint-value` is a variant**: In WIT, `paint-value` is `variant { flag(bool), scalar(f32), tool-index(u32) }`. In the `wall-feature-flag.custom` list, the value type is `paint-value`. The converter must correctly encode/decode each `PaintValue` variant.
- **Determinism**: `WallFeatureFlags.custom` map must round-trip in deterministic order. Use sorted-by-key encoding to ensure repeated runs produce identical WIT bytes.
- **The macro's `build_*_world_glue` functions generate inline WIT for `wit_bindgen::generate!`**: After Packet A, these use `include_str!` pointing to the canonical disk files. Changing the disk files automatically propagates to the generated bindings.

## Code Change Surface

### Selected approach

Step 1 (WIT types): Modify the canonical disk WIT files directly:
- `wit/deps/types.wit`: `enum extrusion-role { ..., custom }` → `variant extrusion-role { ..., custom(string) }`
- `wit/deps/ir-types.wit`: `enum paint-semantic { ..., custom }` → `variant paint-semantic { ..., custom(string) }`
- `wit/deps/ir-types.wit`: `record wall-feature-flag` — add `custom: list<tuple<string, paint-value>>` field (wasmtime supports tuple in list; currently `wall-feature-flag` has no `custom` field; this is new)

Step 2 (Macro converters): Update the converter functions in `lib.rs`:
- `__slicer_ir_role_to_wit`: match IR variant `Custom(s)` → WIT variant `custom(s)`. Remove `String::new()` synthesis for Custom variants.
- `__slicer_wit_semantic_to_ir`: match WIT variant `custom(s)` → IR variant `Custom(s)`. Remove `String::new()` synthesis.
- `__slicer_ir_feature_to_wit`: encode `custom: HashMap<String, PaintValue>` as `custom: Vec<(String, PaintValue)>` sorted by key for determinism
- `__slicer_wit_feature_to_ir`: decode `custom: Vec<(String, PaintValue)>` (wasmtime bindgen generates tuples as `(K, V)` Rust tuples) → IR `HashMap::from_iter(entries)`
- `ir_to_wit_paint_semantic` and `__slicer_ir_semantic_to_wit`: add arms for `Custom(s)` that route through the WIT variant

Step 3 (Host converters): Update host-side converters in `wit_host.rs` for:
- `ExtrusionRole`: decode WIT `custom(s)` → IR `Custom(s)`
- `PaintSemantic`: decode WIT `custom(s)` → IR `Custom(s)`
- `WallFeatureFlags`: decode `custom: list<CustomEntry>` → IR `HashMap::from_iter(entries.iter().map(|e| (e.key, e.value)))`

Step 4 (Tests): Add three round-trip test cases in `macro_all_worlds_roundtrip_tdd.rs`:
- `extrusion_role_custom_payload_roundtrip`: create IR `ExtrusionRole::Custom("test@1")`, convert IR→WIT→IR, assert payload = "test@1"
- `paint_semantic_custom_payload_roundtrip`: create IR `PaintSemantic::Custom("com.example/test@1")`, round-trip, assert payload preserved
- `wall_feature_flags_custom_payload_roundtrip`: create IR `WallFeatureFlags { custom: HashMap from [("key", PaintValue::Scalar(0.5))] }`, round-trip, assert map has one entry with correct value

### Rejected alternatives that were considered and why they were not chosen:

- **Use `option<string>` instead of `string` for custom variant payload**: An `option` would allow `null`/absent but we need to carry an actual mandatory string. The format `module-id/semantic-name@major` is required by the IR spec. `string` is correct.
- **Use `list<string>` for `WallFeatureFlags.custom` values only**: Would lose the `PaintValue` type information. The custom map is `HashMap<String, PaintValue>`, so the WIT representation must carry both key and value types.
- **Change IR `HashMap` to `Vec` in IR**: Would require IR schema version bump and potentially break existing serialized IR files. The IR is fine — only the WIT boundary conversion needs fixing.
- **Skip `WallFeatureFlags.custom` at this stage**: DEV-016 explicitly calls out all three as loss points. Skipping one would leave the packet incomplete.

## Data and Contract Notes

- IR schema: No change needed to `ExtrusionRole::Custom(String)`, `PaintSemantic::Custom(String)`, or `WallFeatureFlags.custom: HashMap<String, PaintValue>`. These already exist in the IR.
- WIT boundary: The three type changes are additive for the variants (`custom(string)` instead of `custom` arity-0). The `wall-feature-flag.custom` field is new. Existing non-custom variants remain unchanged.
- `paint-value` variant: In WIT `variant { flag(bool), scalar(f32), tool-index(u32) }`. In IR `PaintValue::Flag(bool)`, `PaintValue::Scalar(f32)`, `PaintValue::ToolIndex(u32)`. The existing converter for `PaintValue` should be unaffected since we're only adding a new use of it inside the `wall-feature-flag.custom` list.
- Determinism: The `wall-feature-flag.custom` list encoding must be sorted by key to ensure deterministic round-trip. Document this invariant.

## Locked Assumptions and Invariants

- The `PaintSemantic::Custom(String)` format is `<module-id>/<semantic-name>@<major>` as documented in `docs/02_ir_schemas.md`. This format is preserved as-is, no validation changes.
- The `ExtrusionRole::Custom(String)` format is an arbitrary string chosen by module authors. No format validation is done at the boundary.
- `WallFeatureFlags.custom` is used by paint-region modules to annotate wall segments with custom data. The module ID in the key must match the registering module's ID. This invariant is preserved.
- `wit-bindgen` generates the same Rust type for `variant { custom(string) }` as for `enum { custom }` in the non-custom cases — only the custom variant changes. Non-custom `ExtrusionRole` and `PaintSemantic` round-trip is unaffected by this change.

## Risks and Tradeoffs

- **Breaking WIT change**: Changing `enum` to `variant` and adding a field to `wall-feature-flag` is a WIT-breaking change. All modules compiled against the old WIT will need recompilation. This is expected and acceptable — DEV-016 has been open long enough that no released module uses the old WIT.
- **wasmtime supports `list<tuple<K,V>>`**: Unlike plain WIT which has no anonymous tuples, wasmtime's component model bindgen accepts `list<tuple<string, paint-value>>` as the representation for `WallFeatureFlags.custom`. This is simpler than the `list<record { key, value }>` approach originally planned. Use `list<tuple<string, paint-value>>` in the WIT file.
- **Macro converter update complexity**: The macro converter functions are hand-written trait impls. Updating them from `enum { custom }` arity-0 to `variant { custom(string) }` arity-1 is straightforward but must be done carefully to not break the other variants.
- **Host converter update**: The host converters in `wit_host.rs` are generated by `bindgen!`. After Packet A consolidation, they use `include_str!` pointing to the canonical disk files. When the disk files change, the generated types change and the hand-written host converter code may need updating to match the new generated types.
- **Packet A dependency**: This packet builds on Packet A's canonical WIT source. If Packet A is not complete, the `include_str!` paths may not exist. Run Packet A steps 1-5 first, or confirm `wit/` files are in their final consolidated state.
