# Troubleshooting

- **Prompt too broad:** narrow to one remediation slice and explain the cut before generating.
- **Task mapping unclear:** delegate a `LOCATIONS` search over `docs/07`; present candidates and confirm one via `AskUserQuestion`.
- **No relevant `docs/07` task:** stop; the request is outside the canonical backlog.
- **Another packet active:** keep the new packet `draft` and report the conflict.
- **No OrcaSlicer reference:** state that no OrcaSlicer dependency was found; never invent one.
- **Packet directory exists:** ask whether to overwrite, revise in place, or choose another slug.
- **Approaching the 120k read budget:** stop populating files; emit partial work and a numbered fresh-session handoff.
- **Step adds a struct field or schema/version constant but does not list the struct-literal or test-assertion blast radius in "Files allowed to edit":** the implementation worker will report PARTIAL when `cargo check --workspace --all-targets` fails on test files. Re-author the step with the blast-radius list (LOCATIONS-dispatched first) inside "Files allowed to edit", and add the test-assertion fallout to the step's verification commands. See `references/templates/implementation-plan.md` Blast-radius discipline.
- **AC verification command targets a test binary that has never driven the asserted behavior:** AC fails the "test binary can do it" check. Either pick a binary that has the end-to-end setup (LOCATIONS-dispatched), or author a shim or test fixture in the same step and document it in `requirements.md` §In Scope. See `references/templates/packet.spec.md` AC verification command rule.

## Lessons from Implemented Packets

- **WIT world-version bump + canonical-name assert sites (packet 131, 2026-07-19):** when bumping a `slicer:<world>@X.Y.Z` package version in `crates/slicer-schema/wit/deps/<world>/<world>.wit`, the test `crates/slicer-runtime/tests/contract/wit_drift_detection_tdd.rs` has two canonical-name asserts (currently `macro_layer_world_package_name_is_canonical` and `canonical_world_layer_<world>_run_<verb>_takes_<noun>`) that hardcode the previous version string. The Step 2 worker missed both; the full contract run only caught them in Step 5. **Rule:** every step that edits a WIT package version line must also include `crates/slicer-runtime/tests/contract/wit_drift_detection_tdd.rs` in its "Files allowed to edit" and run the full contract suite (`cargo test -p slicer-runtime --test contract`) as a verification command, not just the targeted AC commands. The Blast-radius discipline entry above covers this in principle; the lesson is that `wit_drift_detection_tdd.rs` is the canonical example, worth naming.
- **Context-cost estimate validation (packet 131, 2026-07-19):** the `context_cost_estimate` in `packet.spec.md` front matter held at M end-to-end (S+M+M+M+S) and standard band was sufficient. The honest cost was correct *because* every step serialized on shared files (the WIT, the SDK, the host derivation, the contract tests all touch the same dispatch/SDK surface in sequence) — no parallelism was safe. If a packet estimates M but its step graph has independent write surfaces, re-rate it L and split or escalate. The estimate is a constraint on the step graph, not a budget the worker can spend.
- **Baseline-provenance discipline (packet 131, 2026-07-19):** when a packet captures a pre-change baseline (SHA-256 digest, golden output, assertion) in Step 1, the captured value is only meaningful if the build state at capture is recorded or reconstructable. Step 1 captured `263238f8…` for the wedge default-config g-code; the post-implementation tree yielded `8a3b645e…` deterministically, but the drift could not be attributed without inspection of the working tree because no commit SHA, no stash reference, and no list of parallel uncommitted changes accompanied the digest. **Rule:** Step 1 baseline captures must record (a) the exact `git rev-parse HEAD` SHA at capture time, (b) the exact list of uncommitted modifications present at capture, and (c) any toolchain or environment variables that affect determinism (release vs debug, `CARGO_TARGET_DIR`, feature flags). The carve-list file or a sibling `baseline-provenance.md` is a good home; the requirement is that a future reviewer can confirm the captured value is the right pre-packet baseline without reconstructing the capture session.
